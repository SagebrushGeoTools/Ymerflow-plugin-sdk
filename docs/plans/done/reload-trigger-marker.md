# Emit a `.py` reload-trigger marker from `build_frontend` — Plan

## Status: DONE — implemented

## Why this exists

A backend plugin's frontend is compiled by `build_frontend()` into the plugin's
`frontend_dist/` (e.g. `plugins/billing/billing/frontend_dist/`). The host backend
content-addresses that directory **once, at worker startup** (`mount_plugin_assets()`
in the host's `@app.on_event("startup")`) and serves the bundle at
`/plugin-assets/{hash}/` with `Cache-Control: immutable, max-age=31536000`. The browser
only fetches a new bundle when that `{hash}` changes, which only happens when a fresh
backend worker recomputes it.

In dev the backend runs under `uvicorn --reload`, whose watcher (watchfiles) only fires
on **`*.py`** changes. A frontend rebuild rewrites only `.js`/`.json` under
`frontend_dist/`, so uvicorn never respawns the worker, the startup hook never re-runs,
and the backend keeps advertising the *old* content hash — the browser keeps serving the
stale, immutable-cached remote. (Verified with uvicorn's own `FileFilter`: a `.py` path
under a reload-dir → `True`; `frontend_dist/*.js` and `remoteEntry.js` → `False`.)

The fix, at the single build code path all plugins share: have `build_frontend()` write
one tiny **`.py` marker file** into the built `out_dir`. That is the `.py` change event
uvicorn's watcher needs — the worker respawns, `startup_event()` re-runs
`mount_plugin_assets()`, the hash recomputes over the new dist, the `base_url` changes,
and the browser fetches the fresh bundle. No `.js` file can do this; a `.py` can.

## Decisions (resolved with user)

1. **Location of the trigger**: in the SDK's `build_frontend()` (not per-plugin `setup.py`,
   not `install-backend-plugins.sh`), so *every* plugin and *every* rebuild path — including
   a bare `pip install -e <plugin>` — emits it for free.
2. **Keep `--reload`**: this makes the existing `--reload` dev loop actually pick up frontend
   rebuilds; it does not replace it.
3. **Complementary, not total**: this fixes the rebuild-while-backend-is-running path. A
   full `runall.sh` restart that leaves an orphaned worker on `:8000` is a separate
   process-hygiene issue (clean-kill in `runall.sh`), out of scope for this SDK change.

## Design

Write the marker as the **last** file into `out_dir`, after the `dist → out_dir` copytree,
so a respawn triggered during the write reads a *complete* dist. `uvicorn --reload-delay`
(host sets `1.0`) debounces the burst of copy events, so the worker respawns once, after the
copy settles.

- **File**: `out_dir/_reload_trigger.py`
- **Content: constant.** The marker exists only to produce a `.py` filesystem event on each
  rebuild. Constant content means an *unchanged* rebuild does not perturb the directory's
  content hash (`content_address_dir` includes this file), so it never causes a spurious
  cache-miss; when the JS actually changes, the `.js` chunk hashes change the directory hash
  anyway. The file is rewritten (new mtime) every build, which is the event uvicorn needs.
- **Harmless as data**: `frontend_dist/` is `package_data`, not a Python package (no
  `__init__.py`), so the marker is never imported; it is inert bytes that happen to end in `.py`.

## Exact change (`ymerflow_plugin_build/build.py`, inside `build_frontend`)

After the existing dist→out_dir copy:

```python
        # Copy dist/ -> out_dir
        if os.path.exists(out_dir):
            shutil.rmtree(out_dir)
        shutil.copytree(dist, out_dir)
        _log(f"wrote built remote '{remote_name}' to {out_dir}")

        # Reload trigger: emit a *.py file as the final write into out_dir. A dev host
        # backend running under `uvicorn --reload` only watches *.py; a rebuilt JS bundle
        # alone never respawns the worker, so it keeps serving the previous content hash of
        # this dir (assets are immutable-cached). Rewriting this .py each build produces the
        # filesystem event that respawns the worker, re-runs the startup content-addressing
        # hook, and makes the new bundle reachable. Content is constant so an unchanged
        # rebuild does not perturb this dir's content hash.
        with open(os.path.join(out_dir, "_reload_trigger.py"), "w") as f:
            f.write(
                "# Auto-generated by ymerflow_plugin_build.build_frontend.\n"
                "# Presence of a *.py file makes `uvicorn --reload` respawn the dev backend\n"
                "# worker when this bundle is rebuilt, so the host re-content-addresses it.\n"
                "# Not imported; inert build artifact. Do not edit.\n"
            )
```

`os` and `shutil` are already imported at module top; no new imports.

## Testing / rollout

1. Reinstall the SDK (it is installed non-editable): `pip install deps/Ymerflow-plugin-sdk`
   (or `pip install -e` it) so `build_frontend` carries the change.
2. Rebuild a plugin frontend (`pip install -e plugins/billing`) while a `--reload` backend is
   running; confirm the worker respawns and `GET /plugins/me` reports a new `base_url` hash
   matching `content_address_dir(frontend_dist)`.
3. Rebuild again with **no** source change; confirm the hash is **unchanged** (constant marker
   ⇒ no spurious cache-miss) even though a respawn fired.
4. Existing SDK tests still pass (the marker adds one file to `out_dir`; update any test that
   asserts an exact `out_dir` file set).

## Out of scope

- `runall.sh` clean-kill of a stale/orphaned backend, and the "rebuild only if changed"
  stamp in `install-backend-plugins.sh` — those live in the host repo and are tracked
  separately.
